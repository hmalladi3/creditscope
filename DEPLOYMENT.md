# Live deployment runbook

Everything in this repo is ready to deploy ‚Äî Dockerfiles, `render.yaml`, CI, all verified working via `docker compose up`. The only remaining steps require an account on each platform, which I can't create on your behalf. This takes about 10 minutes end-to-end. All three tiers are free, no card required.

There's a one-time circular dependency (backend needs to know the frontend's URL for CORS; frontend needs to know the backend's URL) ‚Äî that's why Render comes before Vercel, and there's a final step to go back and update Render once Vercel's URL exists.

## 1. Neon (Postgres) ‚Äî ~2 min

1. Sign up at [neon.tech](https://neon.tech) (GitHub login is fastest).
2. Create a project named `creditscope`.
3. Copy the connection string from the dashboard. Split it into three env vars for Render below: host+db ‚Üí `DATABASE_URL` (as a JDBC URL, see format below), plus `DATABASE_USERNAME` and `DATABASE_PASSWORD`.
   - Neon gives you something like `postgresql://USER:PASSWORD@HOST/creditscope?sslmode=require`.
   - Convert to JDBC form for `DATABASE_URL`: `jdbc:postgresql://HOST/creditscope?sslmode=require`
   - `DATABASE_USERNAME` = USER, `DATABASE_PASSWORD` = PASSWORD from that same string.

## 2. Render (backend) ‚Äî ~3 min

1. Sign up at [render.com](https://render.com) with GitHub (this also grants Render read access to the repo it needs).
2. New ‚Üí Blueprint ‚Üí select the `hmalladi3/creditscope` repo. Render reads `render.yaml` from the repo root automatically.
3. It'll prompt for the env vars marked `sync: false` in `render.yaml`:
   - `DATABASE_URL`, `DATABASE_USERNAME`, `DATABASE_PASSWORD` ‚Äî from step 1.
   - `CORS_ALLOWED_ORIGINS` ‚Äî leave as `http://localhost:5173` for now; you'll update this in step 4 once the Vercel URL exists.
   - `JWT_SECRET` is auto-generated by Render (`generateValue: true` in `render.yaml`) ‚Äî nothing to enter.
4. Deploy. First build takes a few minutes (Maven build inside the Dockerfile). Once live, note the URL ‚Äî something like `https://creditscope-backend.onrender.com`.
5. Sanity check: `curl https://creditscope-backend-XXXX.onrender.com/actuator/health` should return `{"status":"UP"}`.

## 3. Vercel (frontend) ‚Äî ~2 min

1. Sign up at [vercel.com](https://vercel.com) with GitHub.
2. Add New ‚Üí Project ‚Üí select the `hmalladi3/creditscope` repo.
3. Set **Root Directory** to `frontend` (Vercel auto-detects Vite otherwise).
4. Add environment variable `VITE_API_BASE_URL` = the Render URL from step 2.4 (no trailing slash).
5. Deploy. Note the resulting URL ‚Äî something like `https://creditscope.vercel.app`.

## 4. Close the loop ‚Äî ~1 min

Go back to the Render service ‚Üí Environment ‚Üí update `CORS_ALLOWED_ORIGINS` to the Vercel URL from step 3.5 (exact origin, no trailing slash, e.g. `https://creditscope.vercel.app`). Save ‚Äî Render redeploys automatically. Without this step the frontend will load but every API call will fail CORS preflight (see `docs/specs/auth.md` `AUTH-BE-020` ‚Äî this was a real bug caught by actually running the app in a browser, not by the test suite).

## 5. Verify

- Open the Vercel URL. If Render's free instance has gone idle, the first request cold-starts (up to ~60s) ‚Äî the frontend shows a "waking up the server" message rather than looking broken (see `FE-UI-016`).
- Log in with `admin` / `admin123`, confirm you can create a company and add a rating.
- Update this repo's `README.md` "Live demo" line with the final Vercel URL once confirmed working.

## If something breaks

- **CORS errors in the browser console**: `CORS_ALLOWED_ORIGINS` on Render doesn't exactly match the Vercel origin (check for a trailing slash or `www.` mismatch).
- **Backend `503`/won't start**: check Render's logs ‚Äî most likely a bad `DATABASE_URL` format (must be the `jdbc:postgresql://...` form, not Neon's raw `postgresql://...` connection string).
- **Frontend loads but shows no data**: check `VITE_API_BASE_URL` was set *before* the Vercel build ran (it's baked in at build time, not read at runtime ‚Äî redeploy after changing it).
